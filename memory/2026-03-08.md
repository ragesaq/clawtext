## RageFX: Full Project State (memory checkpoint 2026-03-08)

### What It Is
RageFX is a SteamVR compositor-level post-processing tool. Because not all games support ReShade, RageFX hooks into `vrcompositor.exe` directly via an OpenVR driver + Qt6 QML overlay — so effects apply to any game regardless of ReShade support.

### Architecture
- **`driver_ragefx.dll`** — OpenVR driver; injects into `vrcompositor.exe` via `CreateRemoteThread`/`LoadLibraryW`; hooks D3D11 `CreatePixelShader` + `PSSetShader` vtable (indices 15, 9); applies RCAS sharpening + LUT color grading via MinHook
- **`ragefx_overlay.exe`** — Qt6 QML dashboard overlay; config UI; hot-reloads `config.json` changes; calls `vrpathreg` to self-register driver at startup
- **IPC**: config.json only (`%LOCALAPPDATA%\RageFX\config.json`); driver uses `ReadDirectoryChangesW` for hot-reload
- **Shader ID**: heuristic (first PS >20KB within 8s of init = distortion shader); no hashes, survives SteamVR updates

### Effects (Phase 1, working)
- RCAS sharpening (FSR1 headers, `rcasEnable`/`rcasSharpness`)
- 3D LUT color grading (65³ `.cube` files, `lutEnable`/`lutPath`/`lutBlend`)
- FOV crop (`fovCrop`)

### Phase 2 (planned)
- EASU upscaling pass
- Display shim: vtable hook `IVRDisplayComponent` in vrserver to spoof FOV + render resolution; UV remap in compositor shader to round-trip to physical display geometry

### Repo & Build
- Repo: `github.com:ragesaq/compositor-fx.git`, branch `master`
- Workspace: `/home/lumadmin/.openclaw/workspace/compositor-fx/`
- Build: Windows-only (`build-all.ps1`, MSVC, Qt6 6.8.3 msvc2022_64, WiX 3.14)
- **MCP build server**: `.mcp-servers/ragefx-build/` — 8 tools, 13 phase tracking; VSCode stdio transport currently; HTTP transport being built
- Latest commit: `00fb449` — ragefx_icon.png added to git
- BUILD_COUNTER: 29 (last user-run build); next = 30
- Install: perUser, `%LOCALAPPDATA%\RageFX\`, no UAC required

### MCP Server Improvements (in progress, VSCode building now)

- When complete: assistant can call `start_build`/`watch_build` directly without user relay

### Key File Paths

## OpenClaw: key configuration and state risk surfaces

Source/docs review confirms several sensitive areas in this OpenClaw install: (1) model/provider config and `agents.defaults.models` act as a hard gate, so typos or miswired providers can cause confusing fallback or outright failure and should always be validated; (2) auth profile wiring is deterministic but easy to misconfigure, so critical providers need explicit auth/profile validation; (3) the session store writes atomically, but if the JSON becomes unreadable it can be treated as `{}` and later writes can clobber state, so we should run cleanup/repair checks and keep backups; (4) compaction is mature and should generally be left enabled with conservative thresholds; (5) memory config drift is common — `agents.defaults.memorySearch` is the modern place, and embeddings must point at a working provider/model; and (6) cron is relatively robust, with stale run markers cleared on startup, so most instability risk comes from config drift rather than cron itself.
