## Contractors Showdown Data Platform — PRD v0.1 drafted (2026-03-06 17:53 UTC)

Completed a full PRD for a community data platform for the VR game "Contractors Showdown" (UE5.5). PRD was drafted using Sonnet subagent and posted to Discord #ai-projects forum channel (ID: 1475021817168134144).

**PRD covers:**
- Problem statement: no public wiki, no combat simulator, players rely on vibes
- ProtectiveData system mechanics: cone projection from spine bones, angle-based coverage
- Confirmed armor data (IOTV GEN3, SWAT, IMTV, ApexBK, JPC, SDprotecter)
- JSON schema for versioned data store (vests, ammo, weapons)
- Python extraction pipeline from UE5 pak files using repak/CUE4Parse
- 4-phase feature roadmap: Data Foundation → Combat Simulator → Web Frontend → Community+Automation
- Tech stack: Python + SvelteKit + Three.js + FastAPI + Vercel

**Open questions captured in PRD:**
- ForwardVector/UpVector consistency across vests (needed for accurate 3D visualizer)
- Armor HP interaction with AP checks (binary break vs. degradation)
- Blunt damage formula details
- Helmet bone names + whether they use same ProtectiveData system

ClawText's summarizer policy requires confidence >= 0.70 to auto-promote a summary into the extract buffer. ## RGCS v1.2.390: Zero smoothing on both HMD and controllers — root cause investigation

**User assertion (authoritative):** There was ZERO smoothing happening on HMD OR controllers in v1.2.390. MaxDiag should delay controller movement by several seconds; it did not. Sub-millimeter log deltas are not due to stationary testing — the user knows how to generate movement. The issue is real.

**Confirmed code flow:**
- `applySmoothingUnsafe()` is gated by `m_ipcConfigReceived` (deferred init guard)
- If `m_ipcConfigReceived` is false, the function returns immediately — **no smoothing at all**
- `markIpcConfigReceived()` is called in `RGCSDriver.cpp:918` inside the SHM poll block — only fires when `shmSmoothing` data is read
- `m_kimiFiltersInitialized` is only set once `m_ipcConfigReceived` is true AND filters are constructed

**Candidate root cause:** SHM poll block may not be executing / `shmSmoothing.enabled` evaluating to false before `markIpcConfigReceived()` is reached. If IPC config never arrives (or arrives with `strength <= 0.0f`), smoothing stays permanently disabled.

**Critical logic bug found:** In `RGCSDriver.cpp` near line 901:
```cpp
if (shmSmoothing.enabled && shmSmoothing.strength <= 0.0f) {
    shmSmoothing.enabled = false;
}
```
This condition silently disables smoothing if strength is exactly 0. But `markIpcConfigReceived()` is called AFTER this — so if the overlay sends `enabled=true` but `strength=0.0` (initialization race), smoothing is disabled AND IPC config is still marked received. Subsequent strength updates may not re-enable.

**Also confirmed:** `setFilterType()` calls `resetFilters()` unconditionally — if the filter type is ever re-set during operation, all filters are destroyed and must reinitialize from scratch.

**What was changed/committed in this session (v1.2.393):**
1. Added IPC wait warning log (every 1000 frames) in `applySmoothingUnsafe` when `!m_ipcConfigReceived`
2. Added 3000-frame IPC fallback: if IPC config never arrives, force `m_ipcConfigReceived = true` and initialize with defaults
3. Added more verbose `SMOOTHING STATE CHANGED` log entry (includes current strength)
4. NOTE: The edit to `RGCSDriver.cpp` stateChanged block had a syntax error (accidental corruption) — needs review before build

**Build status:** Cannot build on Linux (requires Windows + MSVC/PowerShell). Changes committed as `0de947b` (v1.2.393) but NOT yet built/tested. The inserted IPC fallback patch may also have doubled the `if (!m_ipcConfigReceived)` check due to two sequential edits — needs audit before pushing to build.

**Next actions needed:**
1. Audit `DeviceHandle.cpp` around the `m_ipcConfigReceived` guard — confirm no duplicate/broken blocks from the two edits
2. Fix the `RGCSDriver.cpp` stateChanged log line (syntax error introduced by bad edit)
3. Build on Windows and test with MaxDiag — confirm smoothing now activates
4. Longer term: audit whether the `shmSmoothing.enabled + strength <= 0` early-out is silently blocking smoothing in real sessions
5. Consider whether `setFilterType()` → `resetFilters()` is being called spuriously (every SHM poll frame if type keeps being re-set)
