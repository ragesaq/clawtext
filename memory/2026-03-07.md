# 2026-03-07 — Diagnosis of v1.2.406 "frame-y 15fps" issue

## Problem
v1.2.406 reports "frame-y/slideshow-like visual behavior (~15fps feel)" when HMD smoothing active, even at Recommended strength (~25% slider = 44Hz cutoff). User can't interpret timing, but 3Hz periodic artifact is plausible perception.

## Root Cause: ATW Velocity Prediction Baseline Mismatch

**Critical discovery:** Velocity was computed from **raw** positions while position submitted to SteamVR was **filtered and lagged**.

ATW prediction formula: `future_position = submitted_position + velocity * prediction_time`

When `submitted_position = filtered_position` (lagged ~11ms at 44Hz cutoff) but `velocity = computed_from_raw_position`, ATW predicts from wrong baseline → visual stutter.

### Fix Applied (commit abf1268, v1.2.407)
Compute HMD velocity from **filtered** position instead of raw:

```cpp
// BEFORE (v1.2.406):
const double dvx = rawPos.v[0] - m_lastRawPosition.v[0];  // wrong baseline

// AFTER (v1.2.407):
const double dvx = pose.vecPosition[0] - m_lastSmoothedPos.x;  // filtered baseline

// Same fix for rotation velocity:
const vr::HmdQuaternion_t& qPrev = m_lastSmoothedRot;  // was m_lastRawRotation
const vr::HmdQuaternion_t& qCurr = pose.qRotation;     // was rawRot (same, but comment clarifies)
```

Now ATW prediction uses `future = filtered_position + filtered_velocity * prediction_time`, which cancels out the lag.

## Evidence from User Logs

1. **Telemetry window was cumulative** (samples 6000 → 18500, no reset) → `disableSmoothingOnFailure()` did NOT fire during session → no periodic filter resets.

   - Initial filter seed output was different from immediate prior (possible one-time artifact)
   - User movement within first 6000 samples caused catch-up

3. **No periodic reset pattern** in telemetry confirms the issue wasn't filter resets happening every 227ms (500 samples).

4. **User was stationary** through sample 10000, then moved head at sample 10500 (rawMaxFrameMm jumps to 8.494mm, maxPosDeltaMm to 13.33mm). During stationary hold, posDelta remained sub-millimeter (0.01-0.5mm range).

## Why "frame-y" at 11ms Lag?

At 44Hz cutoff, exponential filter settles to 99.95% in ~11ms. Individual frame delta is small, but:
- ATW prediction_time is typically 11-20ms
- If velocity is wrong, ATW predicts from 11ms-lagged position
- At 3 rad/s rotation, 11ms lag = 0.033 rad = 1.9° error in predicted position
- 1.9° at arm's length ≈ 31mm spatial error → perceptible stutter

The user's "15fps feel" is likely ATW fallback or inconsistent reprojection due to velocity mismatch.

### Critical Struct Bug: m_lastSmoothedPos is NOT .x/.y/.z

`m_lastSmoothedPos` is declared as `vr::HmdVector3d_t` which is defined as `double v[3]` — a plain array. It has NO `.x`, `.y`, `.z` members. The v1.2.407 ATW velocity fix committed as `abf1268` used `.x/.y/.z` accessors, which means it DID NOT COMPILE correctly. Incremental builds silently reused stale object files from v1.2.406, masking this compile error. The ATW fix was therefore never actually running in v1.2.407.

**v1.2.408 commit `e7513ad`**: Fixed all `.x/.y/.z` → `.v[0]/.v[1]/.v[2]` in `DeviceHandle.cpp`. ### Dual-Submission Hypothesis Was WRONG

Investigated whether hook was double-submitting poses. Conclusion: NO.
- `ServerDriverHostHooks` intercepts `TrackedDevicePoseUpdated`, calls `s_driver->onPoseUpdate()` (modifies pose in-place, returns bool), then calls `origFunc()` exactly once if `shouldUpdate == true`.
- `onPoseUpdate()` does NOT call `origFunc()` internally — no re-entrancy.
- Hook is the sole submitter. No fix needed to `ServerDriverHostHooks.cpp`.
- The failed edit attempt (`abf1268` era) also used wrong tool parameter names (`oldText`/`newText` instead of `old_string`/`new_string`).

### Pose Rate "Doubling" Is a Logging Artifact

Static counters inside `onPoseUpdate()` are LOCAL STATICS shared across ALL device calls (HMD + 2 controllers). The apparent ~2170Hz "per-device" rate after smoothing connects is the AGGREGATE of HMD (~1200Hz) + 2 controllers (~650Hz each). Real HMD pose rate is unchanged at ~1200Hz. Not a real doubling.
